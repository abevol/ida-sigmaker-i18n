本文档解释了"为当前函数查找最短唯一签名"搜索背后的数学原理、其与已知字符串算法的关系，以及 Cython `_speedups` 扩展为何使其变得实用。面向的是希望理解搜索*为何*快速（而非仅了解代码*做什么*）的读者。

## 1. 问题定义

数据库 $D$ 是已分析程序各程序段字节的拼接，总长度为 $N$（数十 MB，即 $N \sim 10^7$）。**模式**（pattern） $P = (t_0, t_1, \dots, t_{\ell-1})$ 是一个 token 序列；每个 token $t_j$ 要么是精确字节（值 $v_j$，掩码 $m_j = \text{0xFF}$），要么是通配符（wildcard，掩码 $m_j = \text{0x00}$，匹配任意内容）。$P$ 在位置 $p$ 处**匹配** $D$ 当且仅当：

$$
\forall j \in [0, \ell): \quad (D[p + j] \wedge m_j) = (v_j \wedge m_j).
$$

（其中 $\wedge$ 表示按位与。）令匹配集为：

$$
M(P) = \lbrace p \in [0, N - \ell] : P \text{ matches } D \text{ at } p \rbrace.
$$

本算法仅使用完整精确字节和完整字节通配符：每个 token 掩码 $m_j$ 要么是 `0xFF`，要么是 `0x00`。底层 SIMD 扫描器可以为单独的签名*搜索*功能表示部分半字节掩码，但签名*生成器*以及本文描述的基于索引的搜索既不产生也不依赖部分掩码，并且种子选择器仅锚定在完全精确的字节上。

对于地址 $a$ 处的函数，我们解码其指令，根据通配策略将操作数转换为通配符，得到一个增长中的模式 $P_\ell$（前 $\ell$ 个 token）。我们想要**最短**的 $\ell$ 使得 $P_\ell$ 唯一：

$$
\ell^\ast = \min \lbrace \ell : |M(P_\ell)| = 1 \rbrace.
$$

**范围。** 本文仅关注一个问题：在给定*固定*通配策略下，找到在当前数据库中*唯一*的最短签名。另一个独立且更困难的问题——哪些字节应保持精确以使签名能在重新编译后存活（跨构建鲁棒性）——属于不同的优化领域，刻意不在本文讨论范围之内。通配策略是本算法的输入，而非由其选择的。

## 2. 朴素方法的代价，以及 462 秒花在了哪里

直观方法：对于 $\ell = \ell_{\min}, \ell_{\min}+1, \dots$，扫描整个 $D$ 并统计 $P_\ell$ 的匹配数，当计数达到 1 时停止。即使使用 SIMD 加速的扫描（每次扫描代价 $O(N)$），也需要执行 $L = \ell^\ast - \ell_{\min} + 1$ 次**独立的完全扫描**：

$$
T_{\text{naive}} = O(L \cdot N).
$$

当搜索需要考虑多个锚点（在函数边界外增长，或比较多个候选起始点）时，对于 $A$ 个锚点，复杂度变为 $O(A \cdot L \cdot N)$。在一个真实的 16 MB 模块上，单个函数曾消耗 **462 秒**。整个优化的核心就是将两个乘法因子 $L$ 和 $A$ 从 $N$ 项中移除。

## 3. 单调收缩：一次种子化，然后精炼

匹配集在 $\ell$ 上是**单调非增的**。追加一个 token 就是增加一个约束条件，只会移除位置，不会增加：

$$
M(P_{\ell+1}) \subseteq M(P_\ell).
$$

因此没有理由从头重新计算 $M(P_{\ell+1})$。给定 $M(P_\ell)$，下一个集合只是前一个集合的**过滤**：

$$
M(P_{\ell+1}) = \lbrace p \in M(P_\ell) : (D[p + \ell] \wedge m_\ell) = (v_\ell \wedge m_\ell) \rbrace.
$$

这就是*种子-精炼*（seed-then-refine）递推。我们支付**一个**初始匹配集（"种子"），之后每个后续长度的代价为：

$$
O(|M(P_\ell)|)
$$

与*当前候选计数*成正比，而非 $N$。

这种单调性正是搜索在实践中快速的原因，但仅凭它本身并不能给出无条件的 telescoping 界。如果种子集大小为 $C_0$，种子后有 $R$ 个精炼步骤，单调性仅给出：

$$
\sum_{r=0}^{R-1} |M_r| \le R \cdot C_0.
$$

更强的"近乎 $O(C_0)$"行为是一个经验性或平均情况下的论断。例如，如果每个信息性精确字节在期望中至多保留 $\alpha < 1$ 比例的候选，则期望精炼和为几何级数：

$$
\mathbb{E}\!\left[\sum_{r=0}^{R-1} |M_r|\right]
    \le C_0 \sum_{r=0}^{R-1} \alpha^r
    < \frac{C_0}{1-\alpha}.
$$

通配符字节的选择性 $\alpha = 1$：它们永远不会缩小集合，除了丢弃那些无法再适应数据库右边缘的候选。因此只有精确字节才是*信息性*精炼步骤，昂贵的种子后过滤完全跳过通配符。增量增长循环仍然将新追加的通配符通过同一个原地内核传递，但仅在候选集已经坍塌后维护右边缘边界；这些调用成本低廉，不值得优化掉。通配符密集的模式自然具有更少的信息性步骤 $R$。

有了这个限定，每个锚点的总代价为 $O(\text{seed} + \sum_r |M_r|)$，最坏情况为 $O(\text{seed} + R C_0)$，在真实代码字节上通常小得多。剩下的问题是：**构建种子能有多便宜？**

## 4. 双字节位置索引（计数排序）

通过扫描 $D$ 构建种子仍然是 $O(N)$，并且每个锚点都这样做会使 $A \cdot N$ 的膨胀卷土重来。我们**每次搜索预计算一次**索引，用于回答"这个双字节序列出现在哪里？"，所需时间与答案大小成正比。

对于每个双字节键 $k \in [0, 2^{16})$，定义其桶：

$$
B_k = \lbrace p \in [0, N-1) : D[p] = \lfloor k / 256 \rfloor \ \text{ and } \ D[p+1] = k \bmod 256 \rbrace.
$$

我们将每个窗口起始位置按键分组存储在一个扁平数组 `positions` 中，加上一个 `heads` 偏移数组（CSR / 计数排序布局）：

- `heads[k]` 是桶 $k$ 在 `positions` 中的起始位置；
- 桶 $k$ 为切片 `positions[heads[k] : heads[k+1]]`；
- 桶大小 $|B_k|$ 为 `heads[k+1] - heads[k]`，可在 $O(1)$ 时间内获取。

构建过程是教科书式的**计数排序**，对 $N-1$ 个相邻字节窗口进行，两遍遍历 $D$：

1. **计数。** 对每个窗口键 $k$，递增 `heads[k+1]`。
2. **前缀和** `heads`，使 `heads[k]` 成为桶 $k$ 的起始位置，然后**散列**：使用桶起始位置的副本 $w$，将每个窗口偏移 $i$ 放置在 `positions[w[k]++]`。

$$
T_{\text{build}} = O(N), \qquad S_{\text{build}} = O(N + 2^{16}).
$$

这项工作物有所值，因为构建索引是 $O(N)$ 的，并且可以在**所有**锚点之间复用。有了它，对偏移 $s$ 处包含键为 $k$ 的精确双字节序列的模式进行种子化，仅需：

$$
O(|B_k|): \quad M_{\text{seed}} = \lbrace p - s : p \in B_k, \ \text{pattern fits} \rbrace,
$$

然后针对剩余 token 进行精炼（第 3 节）。完整的 $O(N)$ 种子扫描消失了，取而代之的是桶读取。

## 5. 动态种子选择（1 字节或 2 字节），来自同一个索引

种子代价为 $O(|B_k|)$，因此我们想要**最稀有**的序列。两个改进：

**(a) 选择最小的桶。** 一个模式通常有多个精确双字节序列。选择桶大小最小的那个。

**(b) 单个稀有字节可能胜过常见字节对。** `e8`（`call` 操作码）无处不在；一个单独的 `0f` 后跟一个常见字节，仅凭稀有字节本身可能就更具选择性。因此我们也考虑 **1 字节**序列，并且我们可以从同一个索引中*免费*获得它们。由于键按 $k = (b \ll 8) \mid c$ 排序，且计数排序按键顺序布局桶，所有高位字节为 $b$ 的 256 个双字节键占据 `positions` 的一个**连续**块。因此 1 字节桶是一个 telescoping 和，可简化为一次减法：

$$
\bigl|B^{(1)}_b\bigr| = \sum_{c=0}^{255} \bigl|B_{(b \ll 8)\mid c}\bigr|
= \texttt{heads}[(b{+}1)\ll 8] - \texttt{heads}[b \ll 8],
$$

1 字节候选列表是单个切片 `positions[heads[b<<8] : heads[(b+1)<<8]]`，无需第二个索引、无需额外内存、无需重新扫描。（一个边界修复：位置 $N-1$ 永远不会是双字节窗口的*起始*，因此当它可以产生有效模式起始时，最后一个数据库字节上的 1 字节命中会显式添加。）

种子选择随后在两种宽度的并集上最小化选择性：

$$
(s^\ast, w^\ast) = \arg\min_{(s, w) \in \text{exact runs}} \bigl|B^{(w)}_{\text{key}(s,w)}\bigr|.
$$

由于 1 字节选项是候选种子的一个*超集*，所选种子桶永远不会比纯双字节选择更差：

$$
C_0 = \bigl|B^{(w^\ast)}\bigr| \le \min_{\text{2-byte runs}} |B_k|.
$$

更小的种子意味着更少的候选进入精炼链，这同时改进了最坏情况 $R C_0$ 界和第 3 节所述的期望候选衰减。

选择器刻意仅使用连续的、完全精确的 1 字节和 2 字节锚点。更丰富的种子家族（更长序列、间隔或间隙种子、多上下文种子）被有意*不*使用：这些技术在读取映射中发挥作用，因为不匹配位置在搜索时是未知的，而这里通配符位置在搜索开始前就已由策略确定。种子仅是一个锚点；每个其他精确字节已经被精炼利用，因此更丰富的种子家族只会增加索引复杂度而不会带来新的选择性。仅当性能分析显示种子枚举（而非精炼）成为瓶颈时才值得重新审视（见第 10 节）。

**延迟种子化。** 两种情况跳过索引种子并等待。首先，对于短于 `MIN_USEFUL_SIG_BYTES` $= 5$ 的模式，即使最稀有的序列也过于常见（种子会枚举 $D$ 的很大一部分），因此种子化等待直到模式足够长以具备选择性。其次，当可种子化的前缀全部为通配符时，根本没有精确字节可用于索引键；与其回退到完整的 $O(N)$ 掩码扫描，种子化被延迟直到出现精确字节。两者都避免了物化或扫描一个尚不具备选择性的种子。

## 6. 类型化缓冲区上的原地精炼

每个精炼步骤（第 3 节）都是对候选数组进行过滤。我们将候选表示为类型化的 `uint32` 缓冲区，并使用双指针扫描**原地**压缩幸存者。设读索引为 $r$，写索引为 $w \le r$，目标 $\tau = v \wedge m$：

```
for r in 0 .. count-1:
    c = cands[r]
    if c + j < N and (D[c + j] & m) == τ:
        cands[w] = cands[r]   # keep
        w += 1
return w                      # new count
```

由于始终有 $w \le r$，幸存者在被读取前永远不会被覆盖。缓冲区在种子化时**分配一次**，之后只收缩；大型函数的约 165k 次精炼调用执行了**零分配**。

## 7. 复杂度总结

| 阶段 | 代价 | 频率 |
|------|------|------|
| 索引构建（计数排序） | $O(N)$ | 每次搜索一次 |
| 种子选择 | $O(\ell)$ 遍历 token | 每个锚点 |
| 种子枚举 | $O(C_0)$ | 每个锚点 |
| 精炼，所有信息性步骤 | $O(\sum_r |M_r|) \subseteq O(RC_0)$ | 每个锚点 |

每次搜索：

$$
T = \underbrace{O(N)}_{\text{一次索引构建}} +
    \sum_{\text{anchors}} O\bigl(\ell + C_0 + \sum_r |M_r|\bigr)
$$

对比朴素的 $O(A \cdot L \cdot N)$。$N$ 项只需支付**一次**且被共享；所有逐锚点的数据库级工作都被替换为所选种子桶及其幸存候选上的操作。在最坏情况下，精炼仍可能为 $O(RC_0)$，但在实践中，当精确字节具有选择性时，其衰减接近几何级数。

## 8. 与已知字符串算法的关系

本算法并非文献中的空白地带；其设计与多个经过充分研究的领域密切相关。

精确、无掩码版本的问题本质上等价于**左有界最短唯一子串**（left-bounded shortest unique substring, LSUS）：对于固定的起始位置，找到从该位置开始的最短唯一子串。最近的 LSUS 工作提供了线性时间的后缀数组/LCP 基线，并且最短唯一/缺失子串（SUS/SAS）算法仍然活跃，尤其是在紧凑小字母表字符串上。这些是无通配符情况以及答案长度合理性检查的适当参考点。

带掩码的情况连接到了**通配符模式匹配**（wildcard pattern matching）和**带通配符的最长公共扩展**（longest common extensions with wildcards），其反复出现的经验与我们依赖的相同：锚定在信息性的非通配符位置上，而非统一对待所有位置。**内部模式匹配**（internal pattern matching）是在固定文本中需要亚线性重复子串查询时的自然原语，而非每次搜索重建索引。

最接近的实际邻居来自同一领域。**YARA** 的基于原子的扫描选择一个规则中短小、稀有、非通配符的子串，找到其出现位置（经典方法通过 **Aho-Corasick** 多模式匹配），然后在每个命中处验证完整的掩码模式。这就是二进制签名的种子-精炼。此处的思想类似，但方向相反：它不是匹配已知模式，而是*增长*出最短的唯一模式，使用字节窗口索引作为原子预言机，使用单调原地精炼作为验证器。

生物信息学中的种子设计文献（间隔种子、间隙种子、采样种子和多上下文种子）并**不**能直接迁移，原因在第 5 节中已说明：它对*未知*的不匹配位置进行对冲，而这里的通配符位置在搜索前已经已知。

## 9. 创新点

这是一个**新颖的应用**。文献中具备各个单独的原语，但为解决*这个*问题——增长在实时数据库中最短的掩码字节唯一签名——的组合方案并未在任何地方预先打包。**关键用例**很具体：逆向工程师在二进制文件重建版本间进行交互式函数重定位。这就是将 7.7 分钟的搜索变为数秒的原因（第 2 节）。

我们并**未**声称提出了关于最短唯一子串或通配符匹配的新的一般性理论；各个原语（倒排字节桶、种子/过滤/验证、单调候选过滤）本身都是标准技术。贡献在于特化，以及这些部分在掩码函数签名上的低成本组合：

1. **1 字节索引是免费的。** 这是其中不那么显而易见的部分。一个对相邻字节*对*进行的单一计数排序布局产生精确的 2 字节桶，并且由于桶按键顺序存储，每个 1 字节桶只是同一个 `heads` 数组的连续*边际*：一个范围视图，无需第二个索引，无需额外内存。

2. **从同一结构中实现混合宽度种子选择。** 动态种子选择（Dynamic Seed Selection）通过桶大小比较完全精确的 1 字节和 2 字节锚点，并选择当前可用的最具选择性的种子，因此进入精炼的候选集尽可能小。

3. **单调原地精炼。** 一旦种子化，候选偏移存储在一个仅会收缩的 `uint32` 缓冲区中；精炼处理存活的候选，而非为每个长度重新扫描数据库。

4. **针对逆向工程的专门适配。** 该索引每次搜索的构建和丢弃成本远低于后缀家族结构，且远快于重复完全重新扫描——这正是交互式 IDA 工作流实际所需的。

## 10. 未来的算法方向

当前实现有意保持保守。本节过去要求进行的性能分析工作已经完成：种子桶大小 $C_0$、信息性精炼步骤 $R$、通配符密度以及候选衰减曲线。它将工作重定向了：实际开销在于一个 Python 种子枚举循环（现在已移至 C，第 12 节）和一个全通配符前缀上不必要的完全扫描（现在已延迟，第 5 节），而非更花哨的种子。更丰富的种子家族（更长连续、间隔或多上下文种子）已被测量并搁置；见第 11 节。剩下的一个开放方向：

1. **精确 LSUS 基线。** 当通配符被禁用，或者一个长精确区域主导签名时，后缀数组/LCP LSUS 基线是答案长度和运行时的有用参考。

## 11. 被拒绝的优化

有几个想法在纸面上看起来不错，跳过它们的推理比结论本身更有价值，因此值得记录下来。

两者都经过了同样的处理：对最差的函数进行性能分析，然后构建一个小型对抗性基准测试，刻意构造每个想法的最佳情景，并检查它是否真的胜出。两者都没有，因为性能分析始终指向别处：并非索引构建（约 0.05 秒），并且在种子映射移至 C 后，也不是精炼内核（约 0.1 秒），而是我们当时留下的两件无聊的事情：一个 $O(C_0)$ 循环仍在纯 Python 中逐个装箱整数映射种子候选，以及一个每当模式以全通配符开始时就会触发的完全 $O(N)$ 扫描。修复这些才是改变数字的关键。以下是我们一路上说服自己放弃的内容。

**块精炼。** 显而易见的下一步是将精确字节分组为连续段，并用宽 `uint64` 或 SIMD 加载一次性比较整个段，跳过通配符间隙，其理论依据是更少的指令意味着更少的时间。但基准测试表明并非如此：精炼受限于内存带宽，而非指令数量。它已经是一个紧致的、线性的跨度为 1 的扫描，这是 CPU 流式处理最快的访问模式，因此宽而少的比较对一个等待内存而非 ALU 的循环没有帮助。前提也比看起来更弱，因为昂贵的过滤传递已经跳过了通配符（第 3 节）。精炼时间约在 0.1 秒左右，这里根本没有值得追逐的东西。

**间隔种子交集。** 索引具有一个诱人的特性：每个桶的位置已经排序，因为我们是在一次从左到右遍历数据库的过程中填充它们的。因此对于像 `8B ?? ?? 45` 这样的间隔模式，你可以抓取两个字节桶并用双指针合并取交集。问题在于我们已经精确地做了这件事，只是成本更低：从最稀有的字节种子化然后针对其余字节进行精炼*就是*这个交集，并且它只触及较小的桶。显式合并必须从头到尾读取两个桶，只要其中一个是具有数百万条目的常见字节，这就是严格更多的工作。而且一旦延迟种子化使起始集保持较小规模，合并就是纯粹的开销；基准测试中没有输入达到需要它才能发挥作用的程度。

简而言之，胜利从来不在算法层面。它们是"停止在 Python 中运行这个循环"和"不要扫描一个无法锚定任何东西的前缀的整个数据库"，这是只有通过测量才能发现的东西，而非指向更巧妙的数据结构。微基准测试以 `--check` 模式保留在代码树中，因此如果后续某个更改将瓶颈推回到精炼上，它会大声失败，这两个想法将得到重新评估。

## 12. Cython 如何实现这一切

上述数学在纯 Python 中也是正确的，但在纯 Python 中它不会*快*。热内核（索引构建、种子候选映射和每步精炼）是对数百万字节进行的内存受限、带分支的循环，而这正是 CPython 的逐元素开销（装箱整数、属性查找、解释器调度、边界检查）代价高达 50-100 倍的地方。`_speedups` 扩展将其编译为紧凑的 C 代码：

- **`build_byte_index`** 是第 4 节的计数排序，实现为对 `const unsigned char[:]` 类型化内存视图的 C 循环，在 `nogil` 下运行，使得 $O(N)$ 构建期间 IDA 的 UI 保持响应。

- **`refine_offsets`** 是第 6 节的原地压缩：一个 `nogil` 双指针循环，操作 `unsigned int[:]` 候选视图和 `const unsigned char[:]` 数据视图，无分配。将这一个函数移至 Cython 后，最大测试模块上的精炼时间从 **约 14 秒**（一个被调用约 165k 次的 Python 列表推导式）降至 **约 0.28 秒**，大约 **50 倍**。

- **`seed_offsets`** 是第 5 节的候选映射内核：一个 `nogil` 循环，在 C 语言中将种子桶转换为 `array.array('I')` 形式的模式起始数组（`p - s` 偏移、适配守卫、$N-1$ 边界情况）。这是最后一个留在 Python 中的 `O(C_0)` 循环，一个装箱并遍历整个桶的生成器表达式；将其移至 Cython 后，最坏函数搜索从 **约 12 秒**降至 **约 1 秒**。它与 `refine_offsets` 采用相同的策略，并通过与 Python 版本的交叉校验确保字节级输出一致。

- **`array.array('I')` 是桥梁。** 一个候选集同时是编排层可以切片和返回的一等 Python 对象，*以及* Cython 内部零拷贝的 `unsigned int[:]` 类型化内存视图。同一个缓冲区既是 Python 可见的候选列表，也是 `refine_offsets` 原地压缩的 C 级 `uint32*`，因此候选跨越 Python/C 边界时无需编组且零每次调用分配。这正是第 6 节的"一次分配，只收缩"在整个搜索中成立的保证。

- **`nogil`** 在两个内核上意味着重负载工作在不持有解释器锁的情况下运行，这保持 UI 响应，并为索引不可用时使用的 SIMD 扫描路径留出余量。

当扩展不存在时，`SIMD_SPEEDUP_AVAILABLE` 为 `False`，纯 Python 回退产生**相同**的结果（测试套件中进行交叉校验）；插件仍然工作，只是没有加速。

### 关于 `cimport array` 与 `import array` 的说明

在 `simd_scan.pyx` 中你会看到两者，共享名称 `array`：

```cython
from cpython cimport array       # 编译时：C 级 array.array 类型 + array.clone
import array as py_stdlib_arr_mod        # 运行时：Python array 模块（构造函数）
```

这**不是**冲突或错误；它是文档化的 Cython 惯用写法，用于高效操作 `array.array`，两行代码各司其职。我们给运行时模块起别名 `py_stdlib_arr_mod`，以使每个调用点上的区分一目了然：

- `from cpython cimport array` 仅为**编译时**。它引入 Cython 捆绑的 `cpython/array.pxd` 中的 C 级声明：`array.array` 扩展类型（因此 `cdef array.array x` 是一个静态类型化的 C 变量）以及内联 C 函数如 `array.clone`（分配兄弟缓冲区而不经过 Python 构造函数）。`cimport` 不会创建**运行时名称绑定**。

- `import array as py_stdlib_arr_mod` 是 Python `array` 模块的普通**运行时**导入。它使得*构造函数调用* `py_stdlib_arr_mod.array('I')` 在运行时得以解析（例如 `array.clone` 的模板参数）。

因此每种使用通过名称即可明确无误：`array.*`（`cdef array.array`、`array.clone(...)`）是 cimport 的 C 级 API，而 `py_stdlib_arr_mod.array(...)` 是运行时的 Python 构造函数。它们不再共享名称，因此不存在"覆盖"的问题。（Cython 规范教程显示两行共享名称 `array`；别名化其中一方是相同的惯用写法，只是更明确地写出来了而已。）
